# Как устроен Sketch Preview

Этот документ нужен тем, кто меняет запуск скетчей. Он описывает текущую схему превью: зачем редактор и среда выполнения разделены, какие файлы проходят через конвейер и где сейчас проходят границы реализации.

## Откуда взялась эта схема

В Processing Android Mode скетч в итоге становится Android-проектом: PDE-код превращается в подкласс `PApplet`, среда Processing попадает в приложение, а сам скетч показывается через `processing.android.PFragment`.

У APDE исторически есть похожая идея для быстрого превью: не переустанавливать sketch APK при каждом запуске, а собирать код скетча отдельно и передавать его приложению превью. В APDE 2.0 эта схема адаптирована под scoped storage, современные ограничения на видимость пакетов и явные Android intents.

## Почему здесь два приложения

Редактор и приложение превью разделены намеренно:

| Роль | Пакет |
| --- | --- |
| Редактор APDE | `com.apde2` |
| Sketch Preview | `com.apde2.sketchpreview` |

Редактор не открывает приложению превью весь скетчбук. Вместо этого он делает снимок текущего проекта в своем кэше, собирает нужные артефакты и передает наружу только результаты сборки через `FileProvider` с read grants.

Так UI редактора, хранилище проектов, компиляция и среда выполнения скетча меньше зависят друг от друга.

## Как редактор запускает превью

Для запуска используется явный `Intent`:

| Поле | Значение |
| --- | --- |
| Action | `com.apde2.preview.RUN_SKETCH` |
| Пакет | `com.apde2.sketchpreview` |
| Доп. данные | URI sketch dex, URI data zip, URI dexed libraries, package name, class name, orientation |

Остановка превью идет отдельным broadcast-сообщением:

```text
com.apde2.preview.STOP_SKETCH
```

Логи и ошибки во время работы идут обратно в редактор через:

```text
com.apde2.preview.LOG
com.apde2.preview.CRASH
```

## Путь от проекта до кадра на экране

На уровне кода конвейер выглядит так:

```text
SketchPreviewController
  -> ProcessingAndroidPreviewPipeline
  -> SketchSnapshotter
  -> ProcessingSourceGenerator
  -> AndroidSketchCompiler
  -> SketchPreviewLauncher
  -> PreviewActivity
```

### 1. Снимок проекта

`SketchSnapshotter` копирует текущую папку скетча в:

```text
cacheDir/sketch-preview/snapshot
```

Он находит `.pde`-вкладки, сортирует их в предсказуемом порядке и останавливает запуск, если проект пустой.

### 2. Генерация исходников

`ProcessingSourceGenerator` собирает `.pde`-вкладки в Java-класс, который наследуется от `PApplet`.

На этом же шаге:

- сохраняются пользовательские `import`;
- при необходимости создаются `settings()` и `setup()`;
- копируются `data/`, `assets/`, `code/` и `android/`;
- в сгенерированный класс добавляются мосты для `print`/`println` и ошибок во время работы.

### 3. Компиляция и dex

`AndroidSketchCompiler` сначала компилирует сгенерированный Java-код через встроенный ECJ, а затем передает `.class`-файлы в D8.

На выходе сейчас нужны два главных файла:

```text
out/sketch.dex
out/data.zip
```

`data.zip` появляется только тогда, когда у скетча есть данные для передачи среде превью.

### 4. Запуск в Sketch Preview

`SketchPreviewLauncher` выдает preview-приложению доступ к артефактам через `FileProvider` и запускает `PreviewActivity`.

В `sketch-preview` происходит уже часть, связанная с исполнением скетча:

1. временные данные предыдущего запуска очищаются;
2. dex и ресурсы копируются в приватное хранилище приложения превью;
3. dex загружается через `PathClassLoader`;
4. создается объект сгенерированного `PApplet`;
5. `PFragment` подключает sketch surface к `PreviewActivity`.

## Инструменты сборки на устройстве

Основное приложение упаковывает артефакты, нужные для компиляции:

```text
assets/toolchains/processing-android/android.jar
assets/toolchains/processing-android/processing-core.jar
```

За это отвечает Gradle-задача `prepareSketchPreviewToolchain`:

- `android.jar` берется из установленного Android SDK;
- `processing-core.jar` извлекается из Processing Android Mode release.

При первом запуске компилятора `ToolchainLocator` распаковывает эти assets в приватное хранилище редактора:

```text
filesDir/toolchains/processing-android/
```

## Ошибки и вывод

Ошибки сборки описываются моделью `PreviewDiagnostic(file, line, column, message)`. `PreviewDiagnosticParser` уже разбирает типичные сообщения Java/ECJ и может быть расширен под новые источники diagnostics.

`MainActivity` показывает диагностику в существующей консоли ошибок. Следующий естественный шаг - привязать `file + line + column` к открытым вкладкам редактора и маркерам ошибок у строк.

Во время работы скетча обратно в редактор возвращаются:

- вывод `stdout` и `stderr`;
- необработанные падения;
- собственные сообщения Sketch Preview во время работы.

## Какие файлы проекта учитываются

Снимок и генератор уже сохраняют:

- несколько `.pde`-файлов;
- `data/`;
- `assets/`;
- `code/`;
- `android/`;
- ориентацию из текущих свойств скетча.

## Где реализация пока заканчивается

Текущий Sketch Preview закрывает путь "собрать код и быстро запустить его". Он еще не равен полноценному слою экспорта Android-приложения.

Пока отдельно остаются:

- permissions и полное редактирование manifest;
- поиск и подключение библиотек;
- объединение ресурсов;
- надежный инкрементальный кэш для ECJ/D8;
- сборка, подпись и экспорт готового APK.

Если меняешь этот слой, держи границу между UI редактора, проектным хранилищем, компиляцией и приложением превью. Именно это разделение сейчас делает запуск скетча управляемым.
